# 宝石系统：统一为「金色总数」后端改造清单

> 前端已于 commit `5d0d672` 完成并上线，**向前兼容**（迁移前后都能跑）。
> 本文是配合的**后端**（okenglish.site）改动项。建议**先部署后端，再确认前端**，
> 或两端同时上线。迁移完成后 `gold === total`，其余颜色彻底退化为纯视觉。

## 背景

历史上宝石分五色：`blue / green / red / purple / gold`，各色独立计数，
`total = 五色之和`。商品按指定颜色定价、按指定颜色扣费，导致：

- 用户余额分散在多种颜色，**购物时某一色不够就买不了**（即便总数足够）。
- 奖惩按颜色，体验割裂。

目标：**对用户只呈现一种「金色」总数**；颜色仅在奖励/惩罚动画里做随机视觉效果。

## 前端已做（无需后端动）

- 余额/商店/状态栏/手机图标统一显示**金色总数**（`crystal.total`）。
- 可购性、解锁按 `total` 校验。
- `spend` 按 `total` 校验，实际扣费**金色优先、否则最富颜色**（迁移后必命中金色）。
- 答对即奖、连错即扣，**即时**生效；扣除显示「−N 💎」下落动画，总数立即减少。
- 奖励/惩罚飘字颜色随机。
- 商店购买仍只传 `product_id`，由**服务端权威结算**。

## 后端需要改（okenglish.site）

### 1. 数据迁移（一次性）
把每个用户的余额合并进金色：

```sql
UPDATE user_crystal
SET gold  = blue + green + red + purple + gold,
    blue  = 0, green = 0, red = 0, purple = 0;
```

> 迁移前请备份。迁移幂等：可加 `WHERE blue+green+red+purple > 0` 避免重复执行影响。

### 2. `GET /crystal/state`
- 返回 `gold` 即为总数；`blue/green/red/purple` 返回 0。
- `total` 字段（若有）保持 = 五色之和（迁移后即 = gold）。

### 3. `POST /crystal/earn`
- 忽略请求体里的 `color`，一律累加到 `gold`。
- 返回最新 state（同上）。

### 4. `POST /crystal/spend`
- 忽略请求体里的 `color`，从 `gold`（= 总数）扣减。
- 余额不足才返回 `error`。
- 前端 penalize / 解锁 / 兑换都走这个端点。

### 5. `GET /shop/products`
- 商品价格 `price[].color` 统一改为 `gold`（金额 `amount` 不变）。

### 6. `POST /shop/buy`
- 从 `gold` 扣除商品价格。
- 仍按 `product_id` 服务端计价（前端不传金额，防篡改）。

## 验收

- 迁移后任意老用户：余额全部显示为金色，数值 = 原五色之和。
- 商店里所有商品价格显示金色；只要总数够即可购买。
- 答题：答对加宝石（随机色飘字 + 总数增）；连错扣宝石（−N 下落 + 总数减），即时。
- 解锁手机游戏单元：扣金色总数即可，不再卡某一色。

## 回滚

后端各端点改动是「忽略 color、统一走 gold」，与旧前端不冲突（旧前端按色读，
迁移后非金色都是 0，会显示 0——所以**务必前端一起上线**，避免旧前端显示 0 余额）。
数据迁移不可逆，回滚需从备份恢复五色明细。
